# Roadmap — Booking Service (project-387)

Бэклог для отработки цикла **issue → triage → PR → review → scheduled checks**
с OpenCode-агентом как участником командного процесса. Не про новую
продуктовую функциональность саму по себе, а про инфраструктуру совместной
разработки поверх неё.

Приоритет — сверху вниз. Порядок выбран так, чтобы сначала обкатать сам
процесс на небольших, чётко ограниченных задачах, а затем перейти к более
цельным фичам.

## 1. CRUD-дыры (быстрые, для обкатки цикла)

- [ ] **Редактирование типа события** — `PATCH /event-types/{id}` + форма в
      `AdminEventTypesPage`
- [ ] **Удаление типа события** — `DELETE /event-types/{id}` + подтверждение в
      админке
- [ ] **Отмена бронирования** — `DELETE /bookings/{id}` + кнопка в
      `AdminBookingsPage`

## 2. Пользовательские настройки отображения

- [ ] **Выбор часового пояса вручную** — сейчас таймзона гостя определяется
      только автоматически из браузера; нужен явный выбор/переопределение
- [ ] **Формат времени 24ч / 12ч** — переключатель в UI, влияет на
      `formatDateTime.ts`

## 3. Продуктовые фичи

- [ ] **Буфер между встречами** — настраиваемый интервал (например, 10 мин)
      между соседними слотами одного `EventType`; требует изменения контракта
      (TypeSpec) → полный design-first цикл
- [ ] **Кастомные рабочие часы по типу события** — вместо захардкоженного
      09:00–18:00 UTC, задаётся при создании `EventType`

## 4. Triage

- [ ] **Пограничный случай UTC-окна** — 14-дневное окно бронирования
      считается в UTC, из-за чего в часовых поясах восточнее UTC (например,
      UTC+3) поздно вечером видимое окно "съезжает" на день раньше, чем
      ожидает пользователь. Задокументировано в `decisions-log.md` проекта
      386 как ожидаемое поведение, но не проверено с точки зрения UX. Issue
      без готового решения — агент должен разобраться в коде, решить,
      баг это или нет, и предложить фикс либо обоснованно закрыть.

## Не в этом плане (осознанно отложено)

Из полного списка "куда развивать" (см. `decisions-log.md` проекта 386):
регистрация/аккаунты, интеграции с внешними календарями, уведомления,
повторяющиеся события, аналитика — это отдельные по масштабу эпики,
требующие архитектурных решений (авторизация, БД вместо in-memory store),
выходящие за рамки задачи текущего шага курса.
